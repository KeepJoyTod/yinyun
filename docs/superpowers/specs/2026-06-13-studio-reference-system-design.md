# 影约云门店工作台功能融合设计

## 目标

将 `studio-workbench` 建设为门店员工日常使用的完整工作台。功能结构参考简约网的清晰分组，但继续使用影约云现有深色侧栏、琥珀强调色、统一后端和私有 OSS。

## 产品边界

- `studio-workbench`：门店员工、门店主管使用。
- `admin-ui`：平台管理员和系统运维使用。
- `mobile-uniapp`、`client-web`：客户预约、查单、取片使用。
- 工作台只处理已经进入统一数据库的订单，不创建客户预约。
- 微信、抖音、H5、门店录入订单统一写入 `yy_order`。

## 信息架构

工作台菜单分为：首页、商户、商品、订单、服务、内部协作、资源、会员、工具、营销、统计、设置。

采用完整菜单分期开放：

- 已有真实接口的页面立即接入。
- 尚未实现的页面显示统一建设状态、依赖能力和预计落点。
- 不允许菜单点击后空白、404 或展示虚假业务数据。
- 旧路由保留重定向，避免线上收藏地址失效。

## 交互规范

- 页面使用紧凑的“标题/命令、筛选、内容、详情抽屉”结构。
- 常用筛选直接显示，高级条件折叠；筛选状态写入 URL。
- 批量命令只在选中数据后出现。
- 所有数据页必须提供加载、空数据、错误、无权限和重试状态。
- 写操作必须防重复提交，并明确显示进行中、成功和失败。
- 动效只服务层级和状态反馈，并尊重系统减少动态效果设置。

## 数据与权限

- Spring Boot、RuoYi、PostgreSQL、Redis 和阿里云私有 OSS 保持不变。
- 工作台新增 bootstrap 接口，返回员工、门店范围、权限、功能开关和待办数量。
- 对前端暴露的雪花 ID 使用字符串。
- 管理员可看全量；门店主管只看所属门店；普通员工只看授权模块和门店数据。
- 预约改期和支付占位必须经过库存账本，不能只修改订单时间。

## 分期

1. 完整菜单和通用页面框架。
2. 订单、排期、门店、服务组和库存闭环。
3. 商品、客户、员工和现有客片能力。
4. 工单、会员、卡券、营销和动态表单。
5. 实时报表、财务和业务日志。
6. 全量回归、地图更新、部署与回滚证据。

## 验收

- 所有菜单均可进入有效页面状态。
- 店员工作台不出现创建客户预约入口。
- 订单、排期、客片、商品、客户和员工使用真实统一数据库。
- 抖音、微信和本地订单可统一筛选。
- 不发生预约时段超卖。
- 私有 OSS 裸链接不可访问。
- 桌面常用视口无重叠和横向溢出。
