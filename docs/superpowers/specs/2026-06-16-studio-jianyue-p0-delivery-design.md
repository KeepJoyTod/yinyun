## 目标

在不更换技术栈、不新建第二套预约账本的前提下，把 `studio-workbench` 做成门店可实际使用的工作台 P0，重点完成四个页面能力：

1. `预约订单`
2. `首页 / 经营概况`
3. `今日预约`
4. `同步订单`

交付标准不是“页面能打开”，而是“门店员工能依据真实订单、真实门店、真实时段和真实同步结果完成当天履约动作”。

## 核心设计决策

### 1. 增量收口，不做大重构

本轮采用“增量收口”方案：

- 继续复用现有页面与 store 结构。
- 继续复用现有后端接口。
- 只有在现有接口无法支撑工作台真实聚合时，才新增小范围聚合接口。

不采用以下方案：

- 不重建一套 `studio` 专属订单表。
- 不把工作台改造成完全独立后台。
- 不因为对标简约网而重做路由或视觉系统。

### 2. 简约网对齐的是业务口径，不是像素复刻

本轮对齐的重点是：

- 状态分组逻辑
- 员工处理顺序
- 信息层级
- 页面可操作性
- 空态/错误态/加载态一致性

不是：

- 像素级复刻别人的页面
- 保留无业务价值的装饰区块

### 3. 所有可点击元素必须落到真实动作

门店工作台中的按钮、卡片和列表项必须满足以下约束：

- 要么打开真实页面
- 要么打开真实订单/详情抽屉
- 要么触发真实接口调用
- 要么在当前阶段明确展示“建设中 + 缺失依赖”

禁止保留死按钮、空路由、无意义 CTA。

## 订单与状态口径

### 统一订单口径

工作台只认本地 `yy_order`，不在前端直接请求抖音 OpenAPI。

工作台中的统计、列表、趋势和卡片都基于：

- `appStore.orders`
- `appStore.ledgerOrders`

其中：

- `orders` 偏今日履约视图
- `ledgerOrders` 偏本地账本全量视图

### 状态分组定义

以 `orderOperations.ts` 中的规则为准：

- `全部有效订单`：排除已取消和已退单
- `待服务`：`待确认`
- `服务中`：`已确认`、`拍摄中`
- `已完成`：`已完成`、`选片中`
- `已取消`：`已取消`
- `已退单`：退款或退单完成

该口径必须同时用于：

- 首页状态卡
- 预约订单页 tabs
- 经营概况服务订单分项
- 导出说明文案

## 数据流设计

### 1. 同步订单

手动同步入口仍放在预约订单页。

流程：

1. 调用 `POST /yy/channel/DOUYIN_LIFE/orders/sync`
2. 保存最近一次同步结果到 `appStore.lastDouyinLifeOrderSync`
3. 刷新：
   - `loadDashboardStats`
   - `loadSchedule`
   - `loadDouyinSyncHealth`
   - `loadChannelSyncLogs`
4. 如有必要刷新全量账本订单列表
5. 用 notice 呈现 `created / updated / failed / logid`

要求：

- 同步中禁止重复点击
- 同步失败必须显示错误摘要
- 达到安全闸时明确标记 `SUSPICIOUS`
- 不允许只请求接口但不刷新页面状态

### 2. 首页经营概况

优先使用现有工作台聚合数据：

- `orderStatusStats`
- `trendStats`
- `todaySlots`
- 本地 `ledgerOrders`
- 本地 `bookingInventory`

金额口径：

- 若后端无独立财务统计接口，则允许以前端聚合 `yy_order.amount` 做 P0 兜底
- 但必须明确标识“当前为订单聚合口径”，不能伪装成完整财务账

### 3. 今日预约

数据来源：

- `loadSchedule(date, storeName)`
- `appStore.scheduleItems`
- `appStore.todaySlots`

工位块点击优先按以下顺序定位订单：

1. `orderNo`
2. 本地订单 `backendId`
3. 本地订单显示 ID

### 4. 门店来客绑定

数据来源：

- `/yy/store/list`
- `/yy/channelProductMapping/list?channelType=DOUYIN_LIFE`

页面只做可视化和核对，不负责自动猜归店。

归店规则只认：

- `yy_channel_product_mapping` 唯一命中
- 已确认的标准 POI anchor

## 页面交互设计

### 预约订单页

必须收口为真实处理台：

- 顶部显示当前范围、金额、同步状态
- 快捷卡只用于切换真实筛选
- 异常筛选要能叠加到列表
- 点击订单行或快捷块都能打开详情
- 详情抽屉中的状态推进只能走真实 `updateOrderStatus`
- 改期只能走真实 `rescheduleOrder`

### 首页

必须保持“总览 + 可跳转”的结构：

- 卡片点击进入对应页
- 趋势图/排行点击进入对应订单筛选
- 冲突提醒点击进入库存页
- 快捷入口要么提供真实链接，要么明确提示去工具页生成

### 今日预约

必须以“排期承接”为导向：

- 待确认优先
- 只看冲突
- 工位块可点
- 日期切换即刷新真实数据

### 门店管理

必须减少噪声：

- 删除无价值状态文案
- 保留真实营业状态、订单数、待服务单
- 保留真实来客映射数据

## 错误处理与空态

所有工作台页面统一遵循：

- 加载中：展示骨架或明确 loading 文案
- 接口失败：展示错误态和重试动作
- 无数据：展示业务含义明确的空态
- 无权限：展示限制说明，不伪造空数据

重点：

- 同步失败不能静默
- 门店映射为空要解释是“未维护映射”，不是“无门店”
- 经营概况无财务接口时要说明当前为订单聚合口径

## 验收标准

满足以下条件才算本轮 P0 完成：

1. 预约订单页可作为门店真实处理台使用。
2. 同步订单有真实返回、真实刷新、真实异常提示。
3. 首页经营概况、今日预约和订单列表都基于真实本地账本。
4. 所有主要卡片、行项、工位块都可跳到真实目标。
5. 四个真实门店和 `DOUYIN_LIFE` 映射展示正确。
6. 页面不保留死按钮、假数据区块和无意义运营文案。
7. `npm test`、`npm run build` 和本地浏览器验收通过。
